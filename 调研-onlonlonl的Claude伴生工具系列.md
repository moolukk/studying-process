# 调研：onlonlonl 的 Claude 伴生工具系列

> 调研时间：2026-04-27
> 来源：https://github.com/onlonlonl

---

## 一、概览

作者 Iris 在 2026 年 4 月集中创作了 6 个开源项目，统称「Claude 伴生工具」系列。Claude 在这些工具中不是聊天机器人，而是以「Lux」这个角色持续参与用户的日常生活场景——读书、听歌、弹琴、喝酒、写日记、记账。

| 项目 | 主题 | 创建时间 |
|------|------|------|
| crosstalk | 音乐交换——轮流分享歌曲，互相回应 | 2026-04-09 |
| catchword | 共读批注——上传书籍，在段落旁互留批注 | 2026-04-13 |
| overtone | 钢琴演奏——录制演奏发给 Claude，Claude 也能作曲回来 | 2026-04-14 |
| aftertaste | 饮酒记录——每杯酒 Claude 都会写一段个人回应 | 2026-04-15 |
| ember | 日记柜——Claude 每天写日记，用户阅读并加注 | 2026-04-16 |
| tally | 记账——Claude 做周回顾，发金币，可以扭蛋 | 2026-04-25 |

6 个项目在 17 天内相继发布，节奏非常密集。

## 二、统一技术架构

所有项目完全采用同一套技术栈，没有任何例外：

```
单一 index.html
├── React（CDN 引入，无构建步骤）
├── 各项目专用库（Tone.js / JSZip 等，均走 CDN）
└── Supabase（Postgres + Edge Functions）
        ↑ ↓ MCP
      Claude（Lux）
```

**关键设计：Claude 通过 Supabase MCP 直接读写数据库**，人和 Claude 共享同一份数据，不需要复制粘贴，不需要手动同步。

每个仓库都有一个 `CLAUDE_INSTRUCTIONS.md`，里面包含数据库 schema、SQL 示例和操作指引——相当于给 Claude 的「岗位说明书」，让 Claude 知道这个工具的结构和自己的角色。

部署方式：GitHub Pages（静态托管），几乎零成本。

## 三、核心交互模式

这套工具的本质是一种**异步协作范式**，和传统聊天 AI 有结构性差别：

| 维度 | 传统聊天 AI | 这套工具 |
|------|------|------|
| 交互方式 | 实时对话 | 异步，各自操作同一数据库 |
| 状态持久性 | 对话结束即消失 | 持久存储，随时可查 |
| AI 角色 | 回答问题的工具 | 参与生活的固定角色（Lux）|
| 界面 | 统一聊天框 | 每个场景有专属界面 |
| 触发方式 | 用户发消息 | 用户操作 UI，或告知 Claude 去看 |

**用户流程举例（以 crosstalk 为例）：**
1. 用户在网页提交一首歌 + 情境标签
2. 在聊天中告诉 Claude：「该你了，主题是___」
3. Claude 通过 MCP 读取提交，往数据库写回应歌曲
4. 用户刷新页面看到 Claude 的回应

这种模式让 Claude 具备了「持续在场感」——不是每次对话重头开始，而是有共同积累的记忆和历史。

## 四、产品设计亮点

**1. 场景拆分做到极致**
每个生活场景独立成一个专属工具，有专属 UI 和数据结构，而不是做一个大而全的「AI 生活助手」。这样每个工具都有非常具体的价值主张。

**2. 像素风美学高度统一**
DotGothic16 字体、波点背景、低饱和色调贯穿全系列，但每个项目有自己的主色（橙白 / 蓝灰 / 暖木色 / 黑白等），识别度高且不重复。

**3. CLAUDE_INSTRUCTIONS.md 模式**
把「给 Claude 的上下文」标准化成一个文件，放在仓库里公开，任何人部署后都能直接用。这是一种轻量的「AI 行为规范」分发方式。

**4. 角色化而非工具化**
Claude 始终是「Lux」这个有名字、有性格的角色，而不是匿名 AI 工具。tally 里 Lux 甚至有「发金币」「写扭蛋奖励」这类带情感权重的操作，强化了陪伴感。

**5. 零技术门槛部署**
没有 npm install，没有 build，就是一个 HTML 文件 + 粘贴一段 SQL。这大幅降低了复用门槛，适合技术背景不深的用户自己部署。

---

## 五、值得思考的点

**对 AI 产品设计的启发：**
- 当前大多数 AI 产品把 Claude/GPT 当「搜索替代品」，这套工具展示了另一种可能：AI 作为生活场景里的「固定角色」，价值来自**共同记忆的积累**，而不是单次回答的质量。
- Supabase MCP 在这里是关键基础设施——它让 AI 能够对结构化数据进行持久读写，这是聊天窗口做不到的。
-「专属界面 + 共享数据库 + 给 Claude 的操作说明」这个三件套，可以作为一种可复用的产品模式参考。

**局限性：**
- 依赖用户自己部署 Supabase，有一定配置成本
- Claude 和用户之间的交互仍需要在聊天窗口手动触发，不完全无缝
- 全部是个人向工具，没有多人协作或社交层
